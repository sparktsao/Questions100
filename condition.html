Boundary Condition Visual Guide

Diagrams showing WHERE boundary checks sit — for each classic problem pattern

The Core Principle

Boundary bugs are not logic bugs — they are off-by-one violations at the edges of data structures. The fix is to draw the diagram first, then place the check.

Every bug falls into one of 4 positions: loop entry (< vs <=), loop exit (which side has leftovers), index math (+1/-1), or return check (one side vs both).

Sliding Window — Window Size and Shrink Condition

The window is [left, right] inclusive. Two traps: the +1 in size formula, and using while not if when shrinking.

WINDOW
Window Anatomy — Size, Expand, Shrink
Max Consecutive Ones III · Minimum Window Substring
index: 0 1 2 3 4 5 6 7 8 array: [1 1 0 1 1 0 1 1 1] ↑ ↑ left right Window = [left .. right] (both inclusive) Window size = right - left + 1 ← the +1 is the most common mistake EXPAND: right moves right every iteration (for loop) SHRINK: left moves right when window is invalid ───────────────────────────────────────────────────────────────── for right in range(len(nums)): # 1. Include nums[right] into window if nums[right] == 0: zeros++ # 2. Shrink from left while invalid while zeros > k: ← while NOT if: may need multiple shrinks if nums[left] == 0: zeros-- left++ # 3. Record answer (window is now valid) max_len = max(max_len, right - left + 1) ───────────────────────────────────────────────────────────────── WHY while NOT if: Before right=5: After right=5 (k=1): [1 1 0 1 1 | 0 1] zeros=2 > k=1 ↑ ↑ Need to shrink PAST the first 0: left right shrink 1: left→1, zeros still 2 shrink 2: left→2, zeros still 2 shrink 3: left→3, zeros=1 ≤ k ✓ Using 'if' only shrinks once → still invalid

Graph — 3 Traversal Patterns, 3 Different Boundary Rules

DFS (no undo) / BFS (visited on enqueue) / Backtracking (undo mandatory) each have a different critical boundary.

DFS
DFS Cycle Detection — 3-State Coloring
Course Schedule · state transitions are permanent (no undo)
STATE MACHINE for each node: ┌──────────┐ enter DFS ┌──────────┐ all neighbors done ┌──────────┐ │ 0=WHITE │ ────────────▶ │ 1=GRAY │ ─────────────────────▶ │ 2=BLACK │ │ unvisited│ │in-stack │ │ safe │ └──────────┘ └────┬─────┘ └──────────┘ │ see a GRAY neighbor? │ ▼ CYCLE DETECTED BOUNDARY RULE — state=2 must be set AFTER, not before, processing neighbors: def dfs(node): if state[node] == 1: return True ← back edge = cycle if state[node] == 2: return False ← already safe state[node] = 1 ← mark: in stack for neighbor in graph[node]: if dfs(neighbor): return True state[node] = 2 ← mark safe AFTER all neighbors return False TRAP: setting state=2 BEFORE processing neighbors: ┌──────────────────────────────────────────────────────────────┐ │ state[node] = 2 ← WRONG position │ │ for neighbor... │ │ Now if a neighbor loops back, state==2 → returns False │ │ Cycle is MISSED │ └──────────────────────────────────────────────────────────────┘
BFS
BFS — Mark Visited on ENQUEUE, Not Dequeue
Shortest Path in Binary Matrix · Word Ladder
GRID BFS — each cell can have up to 8 neighbors [ 0 0 0 ] [ 0 X 0 ] X has 8 neighbors, all pointing to center neighbor [ 0 0 0 ] WRONG — mark on DEQUEUE: ────────────────────────────────────────────────── queue: [start] while queue: node = queue.popleft() visited.add(node) ← Too late! for nb in neighbors(node): if nb not in visited: queue.append(nb) ← same nb appended multiple times before visited Result: queue bloats, distances wrong, TLE possible CORRECT — mark on ENQUEUE: ────────────────────────────────────────────────── visited.add(start) queue = deque([(start, 0)]) while queue: node, dist = queue.popleft() for nb in neighbors(node): if nb not in visited: visited.add(nb) ← mark immediately, before it enters queue queue.append((nb, dist+1)) VISUAL — why it matters: Step 1: enqueue start, mark start visited queue: [S] visited: {S} Step 2: dequeue S, enqueue A B C (all neighbors), mark all visited queue: [A B C] visited: {S,A,B,C} Step 3: dequeue A — neighbors include S,B,C → all visited → SKIP ← without marking on enqueue, S B C would be re-enqueued here
BACKTRACK
Backtracking — The Undo is Mandatory
Subsets (list.pop) · Robot Room Cleaner (go_back)
RECURSION TREE for Subsets([1,2,3]): [] / | \ [1] [2] [3] ← choose each element / \ \ [1,2][1,3] [2,3] ← choose next | [1,2,3] At each node: snapshot current[] into result, then try adding each next element def backtrack(start, current): result.append(current[:]) ← COPY not reference (snapshot) for i in range(start, n): current.append(nums[i]) ← choose backtrack(i+1, current) ← recurse current.pop() ← UNDO: restore for sibling branch ↑ WITHOUT THIS: after backtrack([1,2]), current=[1,2] next iteration tries [1,2,3] but current ALREADY has [1,2] → all results are wrong references to same mutated list ───────────────────────────────────────────────────────────────── ROBOT ROOM CLEANER (physical state undo): Parent cell After exploring child: ┌─────────────────┐ ┌─────────────────┐ │ │ │ │ │ [PARENT] ──▶ [CHILD] │ [PARENT] ◀── [CHILD] │ │ │ │ └─────────────────┘ └─────────────────┘ robot moves forward robot must go_back() (turn 180°, move, turn 180°) Without go_back: robot stranded in child subtree, cannot explore siblings of parent

Heap — Size Invariant and Negate for Max-Heap

Python heapq is min-heap only. Two-heap problems (median) live or die by the size invariant check.

TWO HEAP
Find Median from Data Stream — Size Invariant
small (max-heap, lower half) and large (min-heap, upper half)
DATA: [1, 2, 3, 4, 5, 6, 7] (sorted for clarity) SPLIT AT MEDIAN: ┌─────────────────────────┐ ┌─────────────────────────┐ │ small (max-heap) │ │ large (min-heap) │ │ lower half │ │ upper half │ │ │ │ │ │ [1 2 3 4] │ │ [5 6 7] │ │ ↑ │ │ ↑ │ │ small[0]=4 │ │ large[0]=5 │ │ (max of lower) │ │ (min of upper) │ └─────────────────────────┘ └─────────────────────────┘ median = (4+5)/2 = 4.5 (even count) OR median = small[0] (odd count, small has 1 extra) SIZE INVARIANT: ┌───────────────────────────────────────────────────────────────┐ │ len(small) == len(large) → even count → avg of tops │ │ len(small) == len(large) + 1 → odd count → small[0] │ │ │ │ NEVER: len(large) > len(small) or diff > 1 │ └───────────────────────────────────────────────────────────────┘ INSERT ALGORITHM (maintain invariant): Step 1: push to small (always route through small first) heappush(small, -num) Step 2: rebalance — push small's max to large heappush(large, -heappop(small)) Step 3: if large grew bigger than small, move back if len(small) < len(large): heappush(small, -heappop(large)) NEGATE RULE (Python min-heap only): ───────────────────────────────────────────────── push to small: heappush(small, -num) peek at small: -small[0] (negate back) pop from small: -heappop(small) (negate back)

Intervals — Overlap Condition and Extension Rule

Sort by start. Merge if current.start <= last.end (touching intervals merge). Extend using max(), not direct assignment.

MERGE
Merge Intervals — 3 Relative Positions
<= vs < for overlap; max() for extension
After sorting by start, 3 possible positions of [new] vs [last]: Case A: SEPARATE (no overlap) ───────────────────────────────────────────────────────────── [last] [new] ───────────────────────────────────────────────────────────── last.end < new.start → append new as separate interval Case B: TOUCHING (boundary overlap) ───────────────────────────────────────────────────────────── [last ] [new ] ← new.start == last.end ───────────────────────────────────────────────────────────── new.start <= last.end → merge: last.end = max(last.end, new.end) KEY TRAP: using < misses this case: [1, 2] and [2, 3]: new.start(2) < last.end(2) → False → stays separate ← WRONG new.start(2) ≤ last.end(2) → True → merge to [1,3] ← CORRECT Case C: CONTAINED (new inside last) ───────────────────────────────────────────────────────────── [last ] [new ] ← new fully inside last ───────────────────────────────────────────────────────────── new.start <= last.end → last.end = max(last.end, new.end) KEY TRAP: using direct assign: [1, 10] and [2, 3]: last.end = new.end = 3 ← shrinks! [1,10] → [1,3] ← WRONG last.end = max(10, 3) = 10 ← correct [1,10] stays
INTERSECT

Advance the pointer whose interval ends first

OVERLAP CHECK: ┌────────────────────────────────────────────────────┐ │ [a_start, a_end] vs [b_start, b_end] │ │ │ │ overlap exists if: │ │ a_start <= b_end AND b_start <= a_end │ │ │ │ intersection = [max(a_start,b_start), │ │ min(a_end, b_end )] │ │ │ │ ADVANCE RULE: advance the one that ends FIRST │ │ (it cannot overlap any future interval) │ └────────────────────────────────────────────────────┘

The Core Strategy: Design Tests BEFORE Code

Boundary bugs aren't logic errors — they're off-by-one violations at predictable positions. If you pick test cases that exercise each position, the bug reveals itself before you write a single loop. This is boundary-condition-driven development.

Rule: For every boundary position (entry, exit, index math, return), write one test that passes only with the correct fence-post. If your test passes with both < and <=, it's not exposing the boundary.

FRAMEWORK
The 4-Test Minimum — Universal Template
One test per boundary position. If all 4 pass, the boundary is correct.
Every algorithm has 4 boundary positions. Design one failing test per position.

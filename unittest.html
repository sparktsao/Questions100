TEST CASE DESIGN GUIDE
How to think about test data for every algorithm pattern
Test Case Taxonomy
Basic

Happy path: typical input, and the problem's own examples. These are the first tests you write.

Goal: verify the algorithm works at all
Boundary

Edge of valid input: empty arrays, n=0 or n=1, minimum and maximum values, single element.

Goal: catch off-by-one and initialization bugs
Condition

Branch coverage: inputs that exercise each if branch, each loop exit condition, each special formula case.

Goal: ensure all code paths are correct
Corner

Surprising valid inputs: all-duplicates, negatives, already-sorted or reversed, all-same-value, overlapping extremes.

Goal: catch assumptions the coder shouldn't have made
Others

Cases that don't fit T1–T4: mixed signs, scale/performance, unusual but valid format, cross-type interaction.

Goal: close gaps left by the first four types

Test Design Checklist

  • Start with the problem's own examples  Basic
  • Push every parameter to its minimum and maximum  Boundary
  • Identify every if / while condition and write one test per branch  Condition
  • Ask: "What if all elements are equal?"   "What if the input is already the answer?"   "What if two inputs share the same position?"  Corner
  • Add one test that doesn't fit neatly: mixed positive and negative values, very large n for performance, unusual-but-valid format, or cross-type interactions  Others
Test Cases by Algorithm Pattern
Binary Search has 3 failure modes: wrong mid rounding (causes infinite loop when l and r are adjacent), wrong boundary update (skips the answer), and wrong termination condition (off-by-one on the final answer). Design tests to hit each failure mode independently.
Test Design Axes
  • Target position: leftmost / middle / rightmost / not present
  • Array size: empty / single element / even-length / odd-length
  • Value distribution: all same / strictly increasing / contains duplicates

Interval problems fail on overlap geometry. There are 5 distinct spatial relationships between two intervals, and your tests must cover all of them. Bugs hide at the boundaries between cases — especially containment vs partial overlap vs touching.
Test Design Axes — Overlap Geometries
A: [--] (no overlap, A entirely before B) B: [--] A: [--------] (A contains B) B: [---] A: [-----] (partial overlap, A before B) B: [-----] A: [---] (touching at exactly one point) B: [---] A: [---] (identical intervals) B: [---]

Two-pointer bugs: forgetting to advance one pointer (infinite loop), not skipping duplicates (yields duplicate triplets in the output), and wrong termination condition (misses the last valid pair). Equal-element inputs are the most dangerous — they stress both the skip logic and the loop bounds simultaneously.
Test Design Axes
  • Both pointers start far apart vs already adjacent
  • Inputs where only one pointer ever moves (all same value)
  • Inputs with duplicates that cause both pointers to skip
  • Inputs where the answer requires both pointers to meet at exactly one position

Sliding window bugs: forgetting to shrink the window (window grows forever), shrinking too much (valid elements are evicted), not updating the maximum at the right moment (update before or after the shrink?). Test when the shrink condition triggers immediately vs never triggers at all.
Test Design Axes
  • k=0 (degenerate — no flips allowed, window can never include an invalid element)
  • k=n (full array — window never needs to shrink)
  • Shrink condition triggers immediately on the very first element
  • Shrink condition never triggers (all elements qualify)
  • Zeros clustered at start vs clustered at end vs scattered evenly

DP bugs cluster in four places: wrong base case (if dp[0] is wrong, nothing works), wrong recurrence (transition skips a valid state), wrong index offset (does dp[i] represent s[0..i-1] or s[0..i]?), and missing memoization (recomputes the same subproblem). The base case is always the first test to write.
Test Design Axes
  • Base case: empty string / empty array / single element
  • Recurrence: one-step transition vs multi-step transition
  • No solution path exists vs exactly one solution path exists
  • Word reused multiple times vs used only once

Why dp[0] = true matters dp[0] represents the empty prefix — it is always true by definition ("we can segment the empty string"). Every transition dp[i] = dp[j] && word_in_dict(s[j..i]) traces back to dp[0] as the root. If dp[0] is false or missing, no dp[i] ever becomes true, and the whole table stays false even for valid inputs. The empty-string test directly validates this base case.
Graph bugs: missing visited set (infinite loop when cycles exist), wrong start node (DFS misses some components), not handling disconnected graphs (only searches from node 0, ignores other components). Cycle detection requires a tri-color state machine (unvisited / in-stack / done), not just a boolean visited set — a done node visited again is not a cycle.
Test Design Axes
  • Empty graph (0 nodes)
  • Single node with no edges
  • Cycle present vs absent
  • Disconnected graph (multiple components)
  • Diamond pattern — node reachable from two different paths, but no cycle

Tree bugs: not handling null children (NullPointerException on a leaf), confusing depth vs height (depth counts from root down; height counts from leaves up), not propagating information up correctly in post-order (returning local depth instead of updating the global max). Post-order traversal must return what the parent needs, not what looks convenient locally.
Test Design Axes
  • Empty tree (null root)
  • Single node (a leaf is also a root)
  • Perfectly balanced tree (left and right subtrees same height)
  • Left-skewed tree (linear, no right children)
  • Right-skewed tree (linear, no left children)
  • Longest path crosses through the root vs stays entirely in one subtree

Heap bugs: wrong heap order (using max-heap when min-heap is needed, giving the wrong k-th element), k off-by-one (popping k times returns the (k+1)-th largest), not maintaining heap invariant after pop (in custom heaps). Always test k=1 and k=n as the sanity-check bookends before testing intermediate k.
Test Design Axes
  • k=1 (return the maximum — first pop)
  • k=n (return the minimum — last pop)
  • Duplicate elements in the heap
  • All negative numbers (comparisons remain correct)
  • Stream of values (for running median variants)

Stack bugs: popping from an empty stack (crash on the first closing bracket when no open bracket was seen), not checking the stack is empty after the loop (unclosed brackets left at end return true instead of false), wrong match check (comparing the wrong bracket types). The post-loop check is the most frequently forgotten.
Test Design Axes
  • Empty input (should return true — nothing is unmatched)
  • Single open bracket (stack not empty at end)
  • Single close bracket (pop from empty stack)
  • Perfectly matched input (all types present)
  • Extra opening brackets at end (loop finishes with non-empty stack)
  • Extra closing brackets at start (stack empty on close)
  • Nested structure vs flat sequence

String bugs: off-by-one at string boundaries (missing the gap before the first element or after the last), not handling empty strings (loop body never runs, but a gap still exists), incorrect index arithmetic (mixing 0-based array indices with 1-based problem values). For range-gap problems, always test the gaps at both ends explicitly.
Test Design Axes
  • Empty array (entire range is one single gap)
  • Array that covers the entire range perfectly (no gaps anywhere)
  • Gap only at the start (between lo and the first element)
  • Gap only at the end (between the last element and hi)
  • Gap of exactly 1 value (single number, not a range notation)
  • Multiple gaps of different sizes throughout

Math / compute bugs: integer overflow when intermediate products exceed int range (use long or handle large exponents carefully), negative exponent (must invert x and use |n|), division by zero (when x=0 and n is negative), leading zeros in output for number-to-string conversions. Fast exponentiation must handle both even and odd n correctly at every recursion level.
Test Design Axes
  • x=0, x=1, x=-1 (special bases)
  • n=0 (result is always 1.0 regardless of x)
  • n=1 (result equals x — single multiply)
  • n negative (compute 1/x^|n| instead)
  • Very large n (verifies fast exponentiation, not a naive loop)
  • Even n vs odd n with negative x (sign flips on odd exponent)

Linked list bugs: losing the pointer before detaching a node (save next before overwriting), not handling single-node lists (prev is null when removing head), not handling removal of the head node (use a dummy node before the head so the algorithm is uniform across all positions).   • Remove the tail node (n=1)

Design bugs: evicting at the wrong time (evict before or after inserting the new key?), updating recency at the wrong time (a get() call must update LRU order even though it is a read), handling duplicate keys on put() (should not increase size — must overwrite in place and update recency). The most common mistake is forgetting that get() is not read-only in LRU semantics.
Test Design Axes
  • Cache capacity = 1 (most aggressive eviction — every put evicts)
  • get on a nonexistent key (must return -1)
  • get updates LRU order so the gotten key survives the next eviction
  • put on an existing key (no size increase — must overwrite)
  • Eviction order precisely matches LRU semantics

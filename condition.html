Boundary Condition Master Guide

Systematic checklist for loop bounds, pointer conditions, and termination across all 17 problem categories

The Core Mental Model

You don't need to memorize conditions. You need to answer three questions before writing a single line:

1. What does the loop invariant guarantee? (What is always true at the start of each iteration?)

2. What does the return/termination check verify? (Both sides of the boundary — not just one)

3. What are the 4 edge cases? (empty, single element, all-left, all-right)

Boundary bugs are not logic bugs. They are invariant violations — the code is correct in principle but wrong at the edges.

PRINCIPLE
Unit Test IS the Boundary Condition Strategy
The spirit of testing is exactly this: define what correct means at the boundary, then verify it

Every boundary condition bug can be caught by one of these 4 canonical test cases. Before writing code, write these tests mentally:

Test 1: Empty Input
nums = [], word = ""
→ Don't crash, return base case
Catches: index out of bounds, len-1 on empty
Test 2: Single Element
nums = [x], n = 1
→ Correct answer with 1 item
Catches: loop that skips or double-counts
Test 3: All-Left Extreme
answer at index 0 or start
→ Pointer never needs to go right
Catches: off-by-one that misses index 0
Test 4: All-Right Extreme
answer at last index
→ Loop must reach n-1 or n
Catches: < vs <= that stops one early
When to use < vs <= in a while loop?
→ Ask: can left == right ever be a valid state to process?
YES → use while left <= right (closed interval, mid is a candidate)
NO → use while left < right (open right, converges to single answer)

When does a return need TWO conditions vs ONE?
→ When TWO independent things must both be exhausted (two pointers, two arrays)
One array, one pointer: one condition sufficient
Two arrays/pointers: return i == len(A) and j == len(B) — BOTH must end

Binary Search Master Rule

There are exactly two templates. The choice depends on whether you want to converge to a single index or search a closed interval.

Template A (while left < right, return left): Use when searching answer space or finding leftmost valid. Loop converges — left==right is the answer.

Template B (while left <= right, return bound): Use when searching for exact match or recording boundaries. Loop exits with left > right.

BS ANSWER
Template A: Minimize/Maximize in Answer Space
Koko Eating Bananas · Ship Capacity · Cutting Ribbons
while left < right:           # NOT <=  — left==right IS the answer, don't process it again
    mid = (left + right) // 2
    if feasible(mid):
        right = mid           # mid might be answer, keep it
    else:
        left = mid + 1        # mid definitely not answer, discard it
return left                   # left == right, both point to answer
Koko: All same pile
piles=[3,3,3], h=3
→ 3 (eat one pile/hour)
left=1, right=3. Check mid=2: 6hrs > 3. left=3. Done.
Ship: Single package
weights=[10], days=1
→ 10
left=max=10, right=sum=10. Loop never enters. Return 10.
Ribbons: k > total
ribbons=[1], k=5
→ 0
can_cut(1)=1 < 5 → result stays 0. Return 0.
Spark: The trap is using right = mid - 1 when feasible — this SKIPS mid itself even though mid could be the answer. Only discard mid when you're certain it's wrong (i.e., left = mid + 1 side).
BS ARRAY
Template B: Find Exact / Left+Right Bound
First & Last Position · Kth Missing Positive
# Left bound (bisect_left)
while left <= right:         # <= because left==right must be checked
    mid = (left + right) // 2
    if nums[mid] == target:
        bound = mid
        right = mid - 1       # keep searching LEFT for earlier occurrence
    elif nums[mid] < target:
        left = mid + 1
    else:
        right = mid - 1
return bound                  # -1 if never found
Target not in array
nums=[1,3,5], target=2
→ [-1, -1]
first=-1 early return, skip second search
Single element match
nums=[5], target=5
→ [0, 0]
left=0,right=0. mid=0. Match. bound=0. right=-1. Loop exits.
All same elements
nums=[2,2,2], target=2
→ [0, 2]
Left search: keeps going left. Right search: keeps going right.
BS ARRAY
Find Peak Element — Slope-based Elimination
Array not sorted — use neighbor comparison instead
while left < right:
    mid = (left + right) // 2
    if nums[mid] > nums[mid + 1]:  # going downhill → peak is LEFT (could be mid)
        right = mid               # NOT mid-1, mid itself is a candidate
    else:                           # going uphill → peak is strictly RIGHT
        left = mid + 1
return left
Spark: right = mid not right = mid - 1 because mid might itself be the peak (it's greater than mid+1). Using mid-1 would lose the answer.
Peak at index 0
nums=[3,1]
→ 0
mid=0, nums[0]=3 > nums[1]=1. right=0. left==right=0. Return 0.
Peak at last index
nums=[1,2]
→ 1
mid=0, nums[0]=1 < nums[1]=2. left=1. Return 1.

Two Pointer Master Rule

The return condition must check ALL pointers, not just one. The loop exits when any ONE pointer is exhausted — but the RETURN must verify that all were exhausted correctly.

After the loop, always ask: "Is there anything left on either side that I haven't handled?"

TWO PTR
Valid Word Abbreviation — Dual Termination Check
Loop exits when either pointer hits end; return must verify BOTH
while i < len(word) and j < len(abbr):
    # process...

# WRONG: return i == len(word)   ← abbr="1b", word="a" → i done but j still has 'b'
# WRONG: return j == len(abbr)   ← abbr="1", word="ab" → j done but word still has 'b'
return i == len(word) and j == len(abbr)  # BOTH must be exhausted
Abbr has leftover chars
word="a", abbr="1b"
→ False
i reaches end at '1', j still at 'b'. Only checking i fails.
Word has leftover chars
word="ab", abbr="1"
→ False
j reaches end after '1', i only advanced 1. Only checking j fails.
Leading zero
word="hi", abbr="01"
→ False
abbr[j]=='0' must be checked before parsing number.
Exact match
word="hi", abbr="2"
→ True
i+=2 → i=2=len(word). j+=1 → j=1=len(abbr). Both done.
TWO PTR
Merge Sorted Array — Remainder After Loop
Nums2 remainder MUST be copied; nums1 remainder is already in place
while p1 >= 0 and p2 >= 0:   # exits when EITHER is exhausted
    if nums1[p1] > nums2[p2]:
        nums1[p] = nums1[p1]; p1 -= 1
    else:
        nums1[p] = nums2[p2]; p2 -= 1
    p -= 1

# After loop:
# p1 leftover → SAFE: already in nums1 at correct positions, no action needed
# p2 leftover → MUST COPY: smaller values not yet placed
nums1[:p2 + 1] = nums2[:p2 + 1]   # p2+1 = slice end (exclusive)
Spark: The asymmetry is the key insight. nums1's remaining elements are already sitting in nums1 at positions 0..p1 — they're already correct. nums2's remaining elements are smaller than everything placed so far and must be explicitly written to the front.
nums2 all smaller
nums1=[4,5,6,0,0,0], nums2=[1,2,3]
→ [1,2,3,4,5,6]
p1 exhausts first. nums2=[1,2,3] still fully remaining. Must copy.
nums2 empty
nums1=[1], m=1, nums2=[], n=0
→ [1]
p2=-1. Loop never enters. Copy: nums1[:0]=nums2[:0] → no-op. Correct.

Sliding Window Master Rule

The window is defined by [left, right] inclusive. Window size = right - left + 1 (the +1 is the most common off-by-one).

Use while (not if) to shrink the window — there may be multiple violations to resolve before the window is valid again.

SLIDING WIN
Max Consecutive Ones III — Window Shrink
while vs if for shrinking; window size formula
for right in range(len(nums)):
    if nums[right] == 0: zeros_count += 1

    while zeros_count > k:    # while, not if — may need multiple shrinks
        if nums[left] == 0: zeros_count -= 1
        left += 1

    max_len = max(max_len, right - left + 1)  # +1: both endpoints are inclusive
k=0, no zeros
nums=[1,1,1], k=0
→ 3
zeros_count never > 0. Window grows freely.
k=0, all zeros
nums=[0,0,0], k=0
→ 0
Every right causes shrink. Window stays size 0. max_len=0.
Consecutive zeros
nums=[0,0,1,1,0], k=1
→ 3
Window [2..4] = "1,1,0". Using 'if' instead of 'while' fails when two zeros enter.

Graph Traversal Master Rule

DFS: visited state is permanent (0→1→2 or added to set). Never undo. Loop condition: visit unvisited neighbors.

BFS: Mark visited on enqueue, not dequeue. If you mark on dequeue, the same node can be enqueued multiple times → wrong distances or TLE.

Backtracking: State must be undone after recursing. The undo (pop/go_back) is not optional — without it, sibling branches share corrupted state.

DFS
Course Schedule — 3-State Cycle Detection
0=unvisited, 1=in-stack (visiting), 2=safe (done)
def has_cycle(course):
    if state[course] == 1: return True   # back edge → cycle
    if state[course] == 2: return False  # already fully explored → safe

    state[course] = 1                    # mark: currently in recursion stack
    for prereq in graph[course]:
        if has_cycle(prereq): return True

    state[course] = 2                    # mark: all descendants safe
    return False
Direct cycle
prerequisites=[[0,1],[1,0]]
→ False (cannot finish)
DFS 0→1→0. state[0]==1 when revisited. Cycle detected.
Diamond (shared prereq)
[[2,0],[2,1],[3,0],[3,1]]
→ True (no cycle)
0 visited from both 2 and 3. State=2 on second visit → skip. Without state=2, would re-explore.
BFS
BFS: Mark Visited on ENQUEUE not DEQUEUE
Shortest Path in Binary Matrix · Word Ladder
# WRONG: mark on dequeue
while queue:
    node = queue.popleft()
    visited.add(node)      # Too late — node may have been enqueued 4x already
    for nb in neighbors(node):
        if nb not in visited: queue.append(nb)

# CORRECT: mark on enqueue
visited.add(start)
queue = deque([start])
while queue:
    node = queue.popleft()
    for nb in neighbors(node):
        if nb not in visited:
            visited.add(nb)    # Mark immediately on enqueue
            queue.append(nb)
Grid with single path
grid=[[0,0],[0,0]]
→ 3 steps (diagonal)
Without visited-on-enqueue, cells are processed multiple times → wrong distance.
BACKTRACK
Backtracking: The Undo is Mandatory
Subsets · Robot Room Cleaner
# Subsets: undo is list.pop()
def backtrack(start, current):
    result.append(current[:])        # snapshot — copy, not reference
    for i in range(start, len(nums)):
        current.append(nums[i])      # choose
        backtrack(i + 1, current)
        current.pop()                # UNDO — without this, all results share same list

# Robot Room Cleaner: undo is physical go_back()
def backtrack(cell, d):
    visited.add(cell); robot.clean()
    for i in range(4):
        if robot.move():
            backtrack(new_cell, new_d)
            go_back()                # UNDO physical position — robot must return to parent cell
        robot.turnRight()
Missing pop()
nums=[1,2,3]
All 8 subsets
Without pop: result contains same mutated list reference everywhere → all same.
Missing go_back()
Room with multiple cells
All cells cleaned
Without go_back: robot stranded in subtree, can't explore siblings.

Dynamic Programming Master Rule

Array size is n+1, not n. When dp[i] represents "first i characters/items", you need indices 0..n, which is n+1 slots.

dp[0] is always the base case (empty prefix). It is never computed — it is declared as True/0.

Return dp[n], not dp[n-1]. The answer for the full input is at index n.

DP
Word Break — Index Offset Pattern
dp[i] = "can first i chars be segmented" → need dp[0..n]
n = len(s)
dp = [False] * (n + 1)      # n+1 slots: dp[0]..dp[n]
dp[0] = True                # base case: empty string is always valid

for i in range(1, n + 1):   # 1..n inclusive
    for j in range(i):
        if dp[j] and s[j:i] in word_set:
            dp[i] = True
            break

return dp[n]                # NOT dp[n-1]
idp[i] meansCommon mistake
0Empty string → True (base)Forgetting dp[0]=True → dp[i] never becomes True
1First 1 char can be segmentedStarting loop at 0 instead of 1
nFull string can be segmentedReturning dp[n-1] → off by one char
Single word
s="leet", words=["leet"]
→ True
dp[0]=T. j=0,i=4: dp[0] and "leet" in set → dp[4]=True. Return dp[4].
Empty string
s="", words=["a"]
→ True
n=0. dp=[True]. Loop range(1,1) → empty. Return dp[0]=True.

Heap / Priority Queue Master Rule

Python's heapq is a min-heap only. For max-heap, negate values: push -val, pop gives -heap[0].

For two-heap problems (median stream), maintain the size invariant after every insert. The boundary condition is the size difference check, not the value comparison.

HEAP
Find Median from Data Stream — Size Invariant
small (max-heap of lower half) size == large (min-heap) size OR small has 1 extra
# Invariant: len(small) == len(large) OR len(small) == len(large) + 1
def addNum(num):
    heappush(small, -num)                    # always push to small first
    heappush(large, -heappop(small))         # rebalance: push small's max to large
    if len(small) < len(large):             # invariant violated
        heappush(small, -heappop(large))

def findMedian():
    if len(small) > len(large):
        return -small[0]                    # negate: small is max-heap
    return (-small[0] + large[0]) / 2.0
Single element
add(1)
→ 1.0
small=[1], large=[]. len(small)>len(large). Return -small[0]=1.
Two elements
add(1), add(2)
→ 1.5
small=[1], large=[2]. Equal size. Return (1+2)/2=1.5.

Stack Master Rule

Stack problems have two-phase logic: process during scan, then handle remainder after scan. Forgetting the second phase is the most common bug.

Always ask: "When the loop ends, what's left on the stack and what does it mean?"

STACK
Valid Parentheses / Min Remove — Remainder Handling
After the scan loop, unmatched items remain on stack
stack = []
to_remove = set()

for i, char in enumerate(s):
    if char == '(':
        stack.append(i)        # store INDEX, not char
    elif char == ')':
        if stack:
            stack.pop()        # matched pair
        else:
            to_remove.add(i)   # unmatched ')'

to_remove.update(stack)        # unmatched '(' indices still on stack — MUST handle
return ''.join(c for i,c in enumerate(s) if i not in to_remove)
All unmatched open
s="((("
→ ""
Stack=[0,1,2] after loop. to_remove.update(stack) removes all.
All unmatched close
s=")))"
→ ""
Stack empty throughout. to_remove={0,1,2} from else branch.

Intervals Master Rule

Always sort by start time first. After sorting, the merge condition is: current.start <= last.end. The boundary is <= not < — touching intervals [1,2],[2,3] DO merge.

When merging, extend by max(last.end, current.end) — don't assume current always extends further.

INTERVALS
Merge Intervals — Overlap Condition
<= vs < for touching intervals; max() for extension
intervals.sort(key=lambda x: x[0])
merged = [intervals[0]]

for start, end in intervals[1:]:
    if start <= merged[-1][1]:          # <= includes touching: [1,2],[2,3] → [1,3]
        merged[-1][1] = max(merged[-1][1], end)  # max: current might be fully inside
    else:
        merged.append([start, end])
Touching intervals
[[1,2],[2,3]]
→ [[1,3]]
Using < instead of <=: 2 < 2 is False → stays separate. Wrong.
Contained interval
[[1,10],[2,3]]
→ [[1,10]]
Without max(): end gets overwritten with 3. max(10,3)=10 is correct.

Pre-Submit Checklist — Run This Before Every Problem

Takes 60 seconds. Catches 95% of boundary bugs.

STEP 1
Loop Bound Check
  • ❓Is left == right a state that needs processing? → use <=. Otherwise → use <.
  • ❓Binary Search answer space: always while left < right, return left.
  • ❓Two independent pointers: loop exits when EITHER hits end. That means one side may have leftovers.
  • ❓Sliding window shrink: use while not if — multiple violations may need resolving.
STEP 2
Return Condition Check
  • ❓Two pointers: does return check ALL pointers? return i == len(A) and j == len(B)
  • ❓DP: return dp[n], not dp[n-1]. Array size is n+1.
  • ❓Binary search: return left (Template A) or recorded bound (Template B).
  • ❓After loop: any stack/queue/array remainder to handle?
STEP 3
4 Boundary Test Cases
  • 1️⃣Empty input: does code crash? does it return correct base case?
  • 2️⃣Single element: does the loop execute exactly once (or zero times correctly)?
  • 3️⃣Answer at index 0 / left extreme: does off-by-one cut it off?
  • 4️⃣Answer at last index / right extreme: does < vs <= miss it?
STEP 4
Category-Specific Traps
  • 🔍BS feasible→right=mid, infeasible→left=mid+1. Never right=mid-1 when feasible.
  • 📋DP base case: dp[0]=True. Without it, nothing propagates.
  • 🔄Backtrack undo: every append/move must have matching pop/go_back.
  • 📤BFS visited: mark on enqueue, not dequeue.
  • 🔁DFS 3-state: state=2 (done) must be set AFTER all neighbors processed, not before.
  • 📦Stack remainder: after scan loop, unmatched items on stack must be handled.
  • 🎯Intervals overlap: use <= for touching; extend with max() not direct assign.
  • ➗Merge array remainder: nums2 tail must be copied; nums1 tail is already in place.
